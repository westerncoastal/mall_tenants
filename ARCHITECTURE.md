# Unified Architecture Specification for Multi-Tenant E-Commerce

This continuous document defines the headless, serverless multi-tenant design engineered to run hundreds of independent vendor storefronts on a single Server instance. The core architecture relies on an absolute decoupling of asset delivery, computational logic, and database state to eliminate the traditional memory overhead of scaling a platform. 

At the edge, web traffic hits a native Caddy reverse proxy server which dynamically maps incoming wildcard subdomains like `*.platform.com` or custom vendor domains without requiring configuration reboots. Instead of spinning up isolated compute blocks or heavy container runtimes per vendor—which would instantly exhaust a 1GB server—the presentation layer is compiled entirely as a static Vite and React Single Page Application utilizing Shadcn/UI component layouts. Caddy handles these storefront files as flat assets directly out of filesystem memory using Linux kernel optimizations, bypassing Node.js runtime costs completely and maintaining a near-zero RAM footprint. 

When a buyer opens a vendor's store, the static frontend code parses the current URL on initialization and immediately injects it into the HTTP headers of all outbound API calls using an axios client configuration where `headers['X-Spree-Store'] = window.location.hostname`. These lightweight API requests route through Caddy straight to a single, centrally shared Spree Commerce Ruby on Rails engine running on the local host. To survive within the strict 1GB server envelope, this application engine is restricted to a single-process Puma mode via `WEB_CONCURRENCY=0`, capped at a tight threading configuration with `RAILS_MAX_THREADS=3`, and bound to memory allocation limits using `MALLOC_ARENA_MAX=2` to eliminate fragmentation and worker context-switching. 

This central engine acts as a stateless router, reading the incoming `X-Spree-Store` header and automatically scopes all downstream database interactions so that transactions return records matching only that specific tenant. Persistent data processing is shifted off the local server entirely and offloaded over the network to an external Neon Serverless Postgres instance using secure transaction pooling endpoints via `DATABASE_URL` strings appended with `?sslmode=require`. Local volatile cache and quick-lookup session tables are handled by a minimal localized Redis instance bound strictly by aggressive memory eviction rules. 

When an administrative launch event or code push occurs on the master branch, an automated GitHub Actions GitOps workflow triggers an on-demand CI/CD pipeline. The automated runner handles the semantic compilation of the static production frontend, securely transfers the built flat-files to the `/var/www/storefront` directory via an encrypted sync action, logs into the Cherry Server via remote SSH to pull the latest backend source, executes required Rails database migrations directly against the cloud-hosted Neon database, and triggers a safe, zero-downtime reload of the background system backend daemon. 

As an ultimate operational safeguard on the host Debian system, a physical 2GB Linux swapfile is permanently allocated at `/swapfile` and written into the system file systems table to absorb unexpected computational spikes during core updates, ensuring continuous uptime and immunity from local Out-Of-Memory kernel termination. Alternatively, if containerized process isolation is preferred over a native host execution loop, the entire localized backend and local Redis cache run rootless inside a unified Podman network managed implicitly through systemd Quadlet container definitions, maintaining the exact same single-process restrictions and cloud database linkages.
